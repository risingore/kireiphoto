# 👑 Git Commit Message Format（言語非依存中核）

**目的**: 統一された、プロフェッショナルな git commit メッセージフォーマットの定義

---

## 基本フォーマット

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 構造の説明

1. **Header（必須）**: `<type>(<scope>): <subject>`
   - 1行目、50文字以内（推奨）
   - タイプとスコープは英語、説明は日本語

2. **Body（オプション）**: 詳細説明
   - 空行を1行空けて記述
   - 72文字で折り返し（推奨）
   - 箇条書きで記述可能

3. **Footer（オプション）**: メタ情報
   - 空行を1行空けて記述
   - `Closes #123`, `Refs ADR-0012` など

---

## Type（タイプ）

**必須項目**。変更の種類を表す。英語で記述。

| Type | 説明 | 使用例 |
|------|------|--------|
| `feat` | 新機能の追加 | 新機能の追加、新モジュールの追加 |
| `fix` | バグ修正 | 構文エラー修正、ロジック修正 |
| `refactor` | リファクタリング | コード構造の改善、パフォーマンス最適化 |
| `docs` | ドキュメント変更 | README更新、ADR追加、コメント追加 |
| `test` | テスト追加・修正 | ユニットテスト追加、テストケース修正 |
| `chore` | その他の変更 | 依存関係更新、設定ファイル変更 |
| `perf` | パフォーマンス改善 | 処理速度の向上、メモリ使用量の削減 |
| `style` | コードスタイル変更 | フォーマット、インデント修正（機能変更なし） |
| `ci` | CI/CD設定変更 | GitHub Actions、ワークフロー設定 |
| `build` | ビルドシステム変更 | 依存関係の追加・削除、ビルド設定 |

---

## Scope（スコープ）

**オプション項目**。変更の影響範囲を表す。英語で記述。スコープはプロジェクトのディレクトリ構造に合わせて定義する。

| Scope | 説明 | 使用例 |
|-------|------|--------|
| `core` | コア・基盤 | エンジン、ロガー、共通基盤など |
| `api` | API層 | REST/Web API、エンドポイント |
| `module` | 機能モジュール | 特定機能を担うモジュール |
| `config` | 設定関連 | 設定ファイル、設定読み込み |
| `test` | テストコード | tests/ 配下 |
| `docs` | ドキュメント | docs/ 配下 |
| `deps` | 依存関係 | 依存関係マニフェスト・ロックファイル |
| `ci` | CI/CD | ワークフロー、パイプライン |
| `infra` | インフラ | Dockerfile, docker-compose 等 |

**複数スコープの場合**: `core,api` のようにカンマ区切りで記述可能

プロジェクト固有のスコープは各プロジェクトの `commit-message-format.md` で追記する。

---

## Subject（説明）

**必須項目**。変更内容を簡潔に日本語で記述。

### ルール
- 50文字以内（推奨）
- 現在形で記述（「追加する」ではなく「追加」）
- 句読点（。、）は使用しない
- 大文字で始めない（文の先頭は小文字）

---

## Body（本文）

**オプション項目**。変更の詳細説明を日本語で記述。

### ルール
- 空行を1行空けて記述
- 72文字で折り返し（推奨）
- 箇条書きで記述可能
- 「なぜ」この変更を行ったかを説明

---

## Footer（フッター）

**オプション項目**。メタ情報を記述。

| タグ | 説明 | 例 |
|------|------|-----|
| `Closes #123` | Issueをクローズ | `Closes #45` |
| `Refs ADR-0012` | 関連ADRを参照 | `Refs ADR-0012, ADR-0013` |
| `BREAKING CHANGE:` | 破壊的変更 | `BREAKING CHANGE: APIの引数順序を変更` |
| `Co-authored-by:` | 共同作成者 | `Co-authored-by: Name <email>` |

---

## チェックリスト

コミット前に以下を確認：

- [ ] Typeが適切か（feat, fix, refactorなど）
- [ ] Scopeが明確か（影響範囲が分かるか）
- [ ] Subjectが50文字以内で簡潔か
- [ ] Bodyに「なぜ」この変更を行ったかが書かれているか
- [ ] 関連ADRがあれば`Refs ADR-XXXX`を追加したか
- [ ] 句読点（。、）を使用していないか
